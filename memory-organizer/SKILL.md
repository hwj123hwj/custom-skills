# Memory Organizer

## Overview

长期记忆（MEMORY.md）是 Agent 的"外部大脑"，存储决策、配置、经验和长期上下文。但存储过多动态信息会导致记忆臃肿和频繁更新。

本技能提供区分**应该存储**（静态、长期价值）vs **不应该存储**（动态、可自动获取）的原则和示例。

---

## 核心原则

### 应该存储的信息

长期价值、决策驱动、不会频繁变化的配置：

1. **核心原则和决策**
   - 代码分工原则（如：Codex 负责实现，我负责设计）
   - 架构决策（如：为什么选择某工具/方案）

2. **系统配置**
   - 代理、端口、Docker 容器等基础配置
   - 数据库连接信息（Host、Port、Database）

3. **技术经验和教训**
   - 工具选择理由（如：Codex CLI 比 acpx 可靠）
   - API 限流应对策略
   - 故障排查步骤和解决方案

4. **账户信息**
   - GitHub、B站、邮箱等主要账号
   - 敏感信息单独存 secrets.md

5. **Agent 架构**
   - 多 Agent 设计（main vs life-assistant）
   - 飞书账号映射

6. **更新日志**
   - 重要变更的时间线
   - 删除/添加技能记录

---

### 不应该存储的信息

动态变化、可自动获取、易于过时的内容：

1. **已安装技能列表**
   - 问题：每次安装/删除都要更新，维护成本高
   - 替代：`ls skills/` 或读取 skills-lock.json
   - 技能的 SKILL.md 已包含触发词和描述

2. **动态项目状态**
   - 问题：项目下线或不再维护后信息过时
   - 替代：只有持续维护的项目才记录

3. **API 详情和 URL**
   - 问题：API 可能变更，URL 失效
   - 替代：可搜索的文档链接，或工具说明

4. **临时脚本和任务**
   - 问题：一次性任务不值得长期记录
   - 替代：实际脚本文件或 memory/YYYY-MM-DD.md

5. **热榜和排行榜 URL**
   - 问题：URL 结构可能变化，信息可在线获取
   - 替代：需要时再搜索

6. **详细操作步骤**
   - 问题：步骤易过时，具体到工具版本
   - 替代：通用原则或脚本封装

## 清理流程

当发现 MEMORY.md 过于臃肿时：

### 1. 审查分类

逐段检查：
- 这是**决策**还是**状态**？（保留决策，删除状态）
- 这是**长期经验**还是**临时信息**？（保留经验，删除临时）
- 这个信息会**频繁变化**吗？（如果会，删除或简化）

### 2. 删除过时内容

标记为删除的内容：
- ~~已删除技能~~（如果已有更新日志记录）
- ~~不再使用的项目~~
- ~~临时测试脚本~~
- ~~API 详细调用示例~~（除非通用）

### 3. 简化表述

保留但需要简化的内容：
- 将表格改为要点（节省 token）
- 合并重复条目
- 删除"待补充"、"观察中"等占位符

### 4. 更新更新日志

在 `## 更新日志` 中添加：
```markdown
- YYYY-MM-DD: 清理 [X] 部分，删除过时内容
```

## 评判标准

决定某条信息是否值得存入 MEMORY.md 时，问：

1. **3个月后还用得着吗？** → 否，删除
2. **其他 Agent 实例需要知道吗？** → 否，删除
3. **能从工作区自动获取吗？** → 能，删除
4. **这是决策/经验还是配置？** → 配置可能动态，保留决策
5. **这条信息有成本吗？** → 重建/搜索成本高则保留

---

## 维护建议

- **定期审查**：每 1-2 个月清理一次
- **控制篇幅**：目标 < 500 行，否则拆分到 references/
- **优先级**：核心原则 > 经验教训 > 配置 > 项目状态
- **避免冗余**：一处信息只存一次，交叉引用而非复制
