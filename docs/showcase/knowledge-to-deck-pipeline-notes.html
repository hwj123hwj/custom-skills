流水线笔记

这份 deck 不是在讲一个抽象概念，而是在展示这套仓库里一条已经跑通的真实链路：知识先被整理成可判断的候选，再被压成卡片，最后进入真正可展示的网页 deck。

Custom Skills Hub01 / 05
ProblemWhy This Exists
核心矫正

不是把搜索结果排版成 PPT，而是先补一层可判断的知识协议。

Before

知识库原来更像“给人看列表”的搜索结果页，候选只有标题和截断摘要，Agent 很难判断它到底值不值得上 deck。

After

先通过 export / review / brief 把候选结构补齐，再让 deck 流水线只消费那些已经带结论、证据、来源和展示潜力的知识条目。

先解决“有没有足够信息去判断”，再谈“做不做得好看”。

Design Shift02 / 05
PipelineMinimal Loop
最小闭环

这条链路真正做的，只是把知识从“可搜”推进到“可展示”。

Step 01

Export

从知识池导出更完整的候选对象，而不是只看标题和摘要。

Step 02

Review

对候选做 deck 适配度体检，先把噪音和偏题项挡在门外。

Step 03

Brief

把入围内容压成知识卡片和页面结构，不让模板直接吃原始内容。

Step 04

Deck

交给 guizang-ppt-skill 做网页 PPT，让知识变成真正可展示的成果本体。

Recipe Driven03 / 05
SourcesReal Knowledge
真实来源

这份 deck 直接来自仓库内已经沉淀好的 spec，而不是 demo seed。

Knowledge to Deck MVP 定义为什么这条线要先走 MVP，明确什么先不做、什么必须补，以及 export 层的必要性。
Knowledge to Deck Agent Spec 定义 Agent 的输入、输出、卡片结构和失败信号，让 deck 生产链变成可复跑的编排流程。

这一步的意义

一旦 deck 能直接长自 `docs`、`stories`、`specs` 这些真实文档，网站展示的就不只是样板，而是仓库本身持续积累出来的知识精华。
