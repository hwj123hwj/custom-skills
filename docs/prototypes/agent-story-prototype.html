为什么做

Origin

原始问题不是“没有资讯来源”，而是来源太多、噪音太高、平台推荐不可信。你真正想解决的是：在有限时间里，稳定获得高密度、可决策的信息。

当前效果

Working
  • 对 OpenAI、Agent、工具链、工作流变化这类信号已经有不错命中。
  • 已经能过滤掉一部分低价值热点和重复话题。
  • 还需要更多真实反馈来判断“哪些内容真的值得长期沉淀”。

触发方式

Install

这一块故意保留成“接近当前 Agent 详情弹窗”的样子，方便后面判断是独立页还是并回现有 AgentModal。

npx custom-skills install intel-agent --agent

Input Strategy

3
bili feed

默认只看动态和标题摘要，不自动走 ASR，先筛再深挖。

2026-05-16
Following-first 收敛

改成优先读 `twitter feed -t following`、`wx biz-articles --unread`、`bili feed`，不再把微博作为默认主来源。

反馈与待优化

Needs Iteration
  • 还缺更多真实场景下的多轮评测，不然很难判断“过滤得准不准”。
  • B 站内容默认不应直接进 ASR，必须先做严格筛选。
  • 小红书更适合作为“经验贴 lookup”，不是日常主信号源。

下一步

Roadmap
  • 把故事页从静态文案升级成结构化文档驱动。
  • 接入评测结果、失败模式和版本变化记录。
  • 给未来测试者留一个明确的反馈入口和协作说明。

未来的社区反馈位

这里以后可以不是“做社区”那么重，而是先做成“招募测试员”的最小形态：谁测了、测了哪个版本、什么场景、反馈了什么、下一步要改什么。

Tester Feedback #01

“对工具链和 workflow 类内容挺准，但理论型架构更新还是偏多。”

Tester Feedback #02

“B 站如果直接转录会太重，先筛选 UP 主和标题更合理。”

Owner Reflection

“个人能力有限，光靠自己测很难形成足够多的反馈，所以页面设计要天然支持协作和迭代。”

IA
  • 左侧是 Story 列表，只显示可检索信息和状态。
  • 右侧是 Story 详情，只展示当前选中 Agent 的故事与反馈。
  • 以后新增 Stories 时，只是多一份 Markdown 文档，不需要继续手改这个页面结构。
这份 HTML 现在主要用来和正式 Web 保持同一个思路：页面结构基本固定，内容来自 `docs/agent-stories/*.md`，真实站点再通过同步脚本生成 `stories-data.json` 供前端消费。

Deck 是能力产出的展柜

`Stories` 讲的是一个 Agent 为什么被造出来、怎么迭代；`Decks` 展示的则是这套系统最终产出的知识精华。 这里会挂知识卡片集、专题 deck 和未来更适合传播的可视化成果。

为什么这份样板是小的

Intentional
  • 当前知识库里真正适合展示的高质量条目还不够多。
  • 所以第一份样板故意先选最干净的候选，优先验证链路，而不是硬拼大主题。
  • 后面真正值得放首页的 deck，应该来自更成熟的知识条目和更明确的专题。

以后可以怎么长

Next
  • 从“知识卡片集”长到“单主题专题 deck”。
  • 给每份 deck 增加来源知识、关联 agent、迭代版本和反馈记录。
  • 把 `Decks` 正式接进 Web，和 `Stories` 一起形成“过程 + 产物”的双视图。

对 Web 的意义

如果 `Stories` 是项目的成长日志，`Decks` 就是这个系统长出来的成品展柜。两者放在同一个网站里，别人不仅能看见你怎么做，还能看见你做出了什么。

这部分是这次新增的重点：Decks 不再只是文档链接，而是一个和 Stories 同级的内容视图。后面如果正式接站，我会优先按这个方向实现。
